## Phase Boundary

Capabilities required by the OpenClaw memcp plugin. Four primitives: `memcp annotate` command, auto-store ID emission, temporal event time extraction on ingest, and workspace tag scoping. Plus output improvements across CLI commands to give the plugin the best data possible.

<decisions>
## Implementation Decisions

### Annotate Command
- **Tags**: Both append and replace modes. `--tags` appends by default, `--replace-tags` replaces. Append is the common path (plugin adds enrichment on top of auto-store's base tags).
- **Salience**: Both absolute and multiplier modes. `--salience 0.9` sets absolute, `--salience 1.5x` multiplies current stability. Detect by presence of `x` suffix. Max flexibility for callers.
- **Scope**: Single ID only (`--id <id>`). No batch support.
- **Parity**: Both CLI (`memcp annotate`) and MCP (`annotate_memory` tool). CLI is the primary access path; MCP is for completeness.
- **MCP schema**: Tool description should include examples of absolute vs multiplier salience usage. Agent-first — good schema descriptions drive correct usage.

### Auto-store ID Emission
- **Mechanism**: Companion output file. Auto-store writes stored memory metadata to a sibling `.ids.jsonl` file alongside the input JSONL. Fits the existing file-watching pattern.
- **Payload per line**: `{memory_id, role, tags, type_hint, content_preview, created_at, source_line}`. Rich enough for plugin key moment detection without a second `memcp get` call.
- **Filtered messages**: Only successfully stored memories appear in the output file. Filtered/rejected content is not emitted.
- **File location**: Sibling file, same directory, same base name with `.ids.jsonl` suffix. E.g., `conversation.jsonl` -> `conversation.ids.jsonl`.

### Temporal Event Time Extraction
- **Birth year**: `user.birth_year` config value in memcp.toml. Required for resolving relative-age references ("when I was 6"). If not set, relative-age references are stored without resolution.
- **Data model**: New `event_time` column (nullable timestamp) on memories table, plus `event_time_precision` enum column (decade, year, month, day). Separate from `created_at`. Indexable for range queries.
- **Extraction timing**: Hybrid. Regex-based extraction runs inline during store (fast, deterministic, free — catches "in 2019", "when I was 6", "last March"). Optional LLM background worker does a second pass for subtler references ("back in college", "during COVID") on memories where regex found nothing. `event_time` gets updated async by the background worker.
- **Confidence**: Precision enum captures how specific the reference is. "In 2019" = year precision. "Last Tuesday" = day. "In the 90s" = decade.

### Workspace Scoping
- **Model**: First-class `workspace` column on memories table. Nullable — NULL means global (no workspace isolation).
- **Query behavior**: Workspace-scoped queries (`workspace=myproject`) return BOTH workspace-scoped AND global (NULL workspace) memories. Global memories like user preferences are always visible.
- **Activation**: Layered precedence — CLI flag (`--workspace`) overrides env var (`MEMCP_WORKSPACE`) overrides config (`default_workspace` in memcp.toml).
- **Auto-store**: Automatically applies the active workspace to ingested memories when MEMCP_WORKSPACE is set or `default_workspace` is configured. Plugin doesn't need to annotate workspace separately.

### CLI/MCP Output Design
- **Annotate response**: Diff-style output. Returns `{id, changes: {tags_added: [...], tags_removed: [...], salience_before: X, salience_after: Y}}`. Shows exactly what changed.
- **Store response**: Includes `event_time` and `event_time_precision` fields when temporal extraction succeeds. Caller immediately knows a temporal reference was parsed.
- **Search results**: `event_time`, `event_time_precision`, and `workspace` included in default output (when non-null). No --verbose flag needed.
- **Recall `--first`**: Session start flag. When set: (1) inject current date/time for temporal context, (2) include preamble explaining how memory works and available commands, (3) include recalled memories. Regular recall (without --first) returns only memories — no preamble, no wasted tokens.
- **Recall preamble**: Hardcoded sensible default shipped with memcp. Configurable override via memcp.toml for custom wording.
- **Recall truncation**: Smart truncation — return first ~200 chars of content with `...` indicator. Agent can `memcp get <id>` for full content. Saves tokens on large memories. Truncation length configurable.
- **Recall related context**: Each recalled memory includes `{related_count, hint}`. `related_count` is the number of memories sharing at least one non-trivial tag. `hint` is a ready-made search command (e.g., `memcp search --tags decision,project-x`) so the agent knows there's more and can explore. Determination is by shared tags — fast DB query, uses existing tag infrastructure.

<specifics>
## Specific Ideas

- Annotate is the plugin's enrichment layer — auto-store handles base storage, plugin calls annotate to add semantic tags (decision, preference, correction) and boost salience on key moments
- The `.ids.jsonl` companion file pattern mirrors the existing JSONL watcher input pattern — the plugin can tail it the same way
- "When I was 6" + birth_year config is the canonical example for temporal extraction. "In 2019", "last summer", "during COVID" are the other priority patterns.
- Token efficiency matters — the plugin injects recall output into LLM prompts, so every byte counts. Smart truncation and --first preamble control are both driven by this concern.
- Plugin design doc (engram/.planning/memcp-openclaw-plugin-design.md) is the upstream source of truth for how these primitives are consumed

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `temporal.rs` (query_intelligence/temporal.rs): Regex-based temporal parser for search queries. Same pattern-matching approach can be extended for content-time extraction, but targeting different expressions (content says "when I was 6", queries say "last week").
- `upsert_salience()` (storage/store/postgres.rs): Existing salience mutation. Annotate command can call this for absolute salience sets.
- `update_memory` MCP tool (transport/server.rs): Existing memory update path. Annotate can reuse the update pipeline for tag changes.
- `format_memory_json()` (cli.rs:96): Central output formatter for CLI. New fields (event_time, workspace) need to be added here.
- `WatchEvent` / watcher.rs: Auto-store file watcher infrastructure. ID emission writes to the companion file from the same pipeline.

### Established Patterns
- CLI commands use clap subcommands (main.rs) with handler functions in cli.rs (`cmd_store`, `cmd_search`, etc.)
- MCP tools defined as `#[tool]` methods on `MemoryService` in transport/server.rs
- Database migrations in crates/memcp-core/migrations/ (numbered SQL files)
- Config via figment: defaults -> memcp.toml -> env vars (config.rs)
- Background workers run in daemon mode (transport/daemon.rs) with tokio spawn loops

### Integration Points
- `cmd_annotate` new CLI handler in cli.rs, wired from main.rs Commands enum
- `annotate_memory` new MCP tool method on MemoryService
- New migration for `event_time`, `event_time_precision`, `workspace` columns on memories table
- Auto-store pipeline (pipeline/auto_store/mod.rs) needs ID emission after successful store
- Config additions: `user.birth_year`, `default_workspace`, recall preamble override
- `recall` command (cli.rs:1174) needs `--first` flag and truncation support

</code_context>
