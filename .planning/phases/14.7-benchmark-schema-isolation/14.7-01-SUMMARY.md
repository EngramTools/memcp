# Phase 14.7 Plan 01: Benchmark Schema Isolation Summary

Schema isolation for PostgresMemoryStore via `new_with_schema()` + `after_connect` pool hook, with ephemeral benchmark schema lifecycle (`--keep-schema` flag) in the benchmark binary.

## What Was Built

### PostgresMemoryStore schema isolation (postgres.rs)

- Added `schema: Option<String>` field to `PostgresMemoryStore` struct
- New `new_with_schema(database_url, run_migrations, search_config, schema)` constructor:
  - Validates schema name (alphanumeric + underscore only, prevents SQL injection)
  - Creates schema via one-off `PgConnection::connect()` before building the pool
  - Builds pool with `after_connect` hook: `SET search_path TO "{schema}", public` on every connection
  - `public` kept in search_path so pgvector and pg_trgm extensions remain accessible
  - Migrations run through the pool and target the schema via search_path
- New `drop_schema()` method: `DROP SCHEMA IF EXISTS "{name}" CASCADE`
- `new()` and `new_with_search_config()` now delegate to `new_with_schema(..., None)` — no behavior change
- All `from_pool` variants include `schema: None` — fully backward compatible

### Benchmark binary wiring (benchmark.rs)

- Added `--keep-schema` CLI flag with description
- Both `run_longmemeval` and `run_locomo` use `new_with_schema(..., Some("benchmark"))`
- Tracing log at startup: `Using isolated benchmark schema`
- Schema cleanup after all configs complete and CI threshold check:
  - Default: `store.drop_schema().await?` (ephemeral — clean state every run)
  - With `--keep-schema`: retained for post-run inspection
