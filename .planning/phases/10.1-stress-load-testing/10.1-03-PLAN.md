<task type="auto">
  <name>Task 1: Run core load test suite and generate reports</name>
  <files>load_test_results/</files>
  <action>
Ensure Postgres is running (`just pg` or verify Docker container on port 5433).

Run the core load test matrix — three R/W ratios at the default concurrency progression:

1. **Read-heavy (80/20) — the primary production workload:**
```bash
DATABASE_URL=postgres://memcp:memcp@localhost:5433/memcp \
  cargo run --bin load-test -- \
    --rw-ratio 80/20 \
    --output-dir load_test_results/80-20 \
    --save-baseline
```
This runs corpus sizes [100, 1000, 10000] at concurrency [10, 50, 100, 500] in both raw and rate-limited modes.

2. **Balanced (50/50):**
```bash
DATABASE_URL=postgres://memcp:memcp@localhost:5433/memcp \
  cargo run --bin load-test -- \
    --rw-ratio 50/50 \
    --output-dir load_test_results/50-50
```

3. **Write-heavy (20/80):**
```bash
DATABASE_URL=postgres://memcp:memcp@localhost:5433/memcp \
  cargo run --bin load-test -- \
    --rw-ratio 20/80 \
    --output-dir load_test_results/20-80
```

4. After all three profiles complete, create a **summary report** at `load_test_results/CAPACITY_REPORT.md` that aggregates key findings:
   - Consolidation table: `| Profile | Corpus | Concurrency | Raw ops/sec | Rate-limited ops/sec | p95 (raw) | p95 (limited) | Fly Tier |`
   - Breaking points identified (where error_rate > 5% or p95 > 1000ms)
   - Fly.io tier recommendations summary
   - Key observations (e.g., "search degrades linearly with corpus size" or "rate limits add ~50ms p95 overhead")

If any run fails (DB connection issues, etc.), debug the root cause before retrying. Common issues:
- Pool exhaustion at concurrency 500: increase Postgres max_connections in Docker (see RESEARCH.md Pitfall 1)
- Embedding worker interference: verify seed uses `embedding_status = 'done'`
- BM25 returning 0 results: check tsvector trigger fired on bulk insert

Expect the full suite to take 10-30 minutes depending on hardware.

  </verify>
