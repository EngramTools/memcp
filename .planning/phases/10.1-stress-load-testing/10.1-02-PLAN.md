2. Create `crates/memcp-core/src/load_test/client.rs` with:

**`pub async fn run_workload(config: &LoadTestConfig, client: &reqwest::Client) -> Vec<RequestResult>`**

Core concurrent driver logic:
- Create `Arc<Semaphore>` with `config.concurrency` permits to bound concurrent requests.
- Create `Arc<Mutex<Vec<RequestResult>>>` for collecting results (or use `tokio::sync::mpsc` channel for lock-free collection — implementer's choice for performance).
- Generate `config.total_ops` operation slots. For each slot, determine if it's a read or write based on `config.rw_ratio.write_pct()`:
  - `(op_index % 100) < write_pct` -> write operation
  - Otherwise -> read operation
- For write operations, cycle through these endpoints with equal distribution:
  - `POST /v1/store` — body: `{"content": "Load test memory {op_index}", "type_hint": "fact", "tags": ["load-test"]}`
  - `PUT /v1/update/{id}` — update a previously stored memory (use a shared Vec of stored IDs, skip if empty)
  - `POST /v1/annotate` — body: `{"id": "{stored_id}", "tags": ["annotated"]}`
  - `DELETE /v1/delete/{id}` — delete a previously stored memory (skip if no IDs available)
- For read operations, cycle through:
  - `POST /v1/search` — body: `{"query": "load test query {op_index % 50}", "limit": 10}` (50 unique queries for cache variation)
  - `POST /v1/recall` — body: `{"query": "recall query {op_index % 50}", "limit": 5}`
  - `POST /v1/discover` — body: `{"query": "discover patterns", "limit": 5}` (if endpoint exists)
  - `GET /v1/export` — export all memories (if endpoint exists)
- For each operation:
  1. Acquire semaphore permit (`.acquire_owned().await`)
  2. Measure with `tokio::time::Instant::now()`
  3. Send request via `client.{method}(url).json(&body).send().await`
  4. Record `RequestResult { endpoint, status, latency_ms, is_error }`
  5. Permit auto-drops when task completes
- Spawn each op as a `tokio::spawn` task. Collect all handles and `futures::future::join_all`.
- Show progress via `indicatif::ProgressBar` (total_ops length, update after each op completes).
- Track stored IDs for update/annotate/delete operations using `Arc<Mutex<Vec<String>>>`. Parse the `id` field from store response JSON.
- If a request fails (connection error, timeout), record as `is_error: true, status: 0`.
- Use `reqwest::Client` with connection pool size matching concurrency: `reqwest::Client::builder().pool_max_idle_per_host(config.concurrency).build()`.

**Error handling:** Never panic on individual request failure. Log the error, record it in results, continue. The error_rate metric captures this.

**Important:** Use `tokio::time::Instant` (not `std::time::Instant`) for latency measurement across await points.

**CLI structure (clap derive):**
```rust
#[derive(Parser)]
#[command(name = "memcp-load-test", about = "HTTP load test harness for memcp capacity sizing")]
struct Cli {
    /// Corpus size: number of memories to seed before testing
    #[arg(long, default_value = "1000")]
    corpus_size: usize,

    /// Concurrency level: number of simultaneous clients
    #[arg(long, default_value = "10")]
    concurrency: usize,

    /// Read/Write ratio: "80/20" (read-heavy), "50/50" (balanced), "20/80" (write-heavy)
    #[arg(long, default_value = "80/20")]
    rw_ratio: String,

    /// Test duration in seconds (per concurrency level)
    #[arg(long, default_value = "30")]
    duration: u64,

    /// Total operations to issue (overrides duration if set)
    #[arg(long)]
    total_ops: Option<usize>,

    /// Disable rate limits for raw capacity measurement
    #[arg(long)]
    no_rate_limit: bool,

    /// Run extended suite including 100k corpus
    #[arg(long)]
    full: bool,

    /// Save results as baseline for future regression comparison
    #[arg(long)]
    save_baseline: bool,

    /// Path to baseline JSON for regression comparison
    #[arg(long)]
    baseline: Option<PathBuf>,

    /// Number of simulated tenants (project values)
    #[arg(long, default_value = "5")]
    num_projects: usize,

    /// Database URL (defaults to DATABASE_URL env var)
    #[arg(long, env = "DATABASE_URL")]
    database_url: String,

    /// Output directory for reports
    #[arg(long, default_value = "load_test_results")]
    output_dir: PathBuf,
}
```

**Main orchestration flow:**

1. Parse CLI args. Parse `rw_ratio` string to `WorkloadProfile` enum.

2. Connect to Postgres via `sqlx::PgPool::connect(&database_url)`. Set pool max connections to `max(concurrency + 10, 50)` to avoid pool exhaustion (from RESEARCH.md Pitfall 1).

3. Run migrations: `memcp::MIGRATOR.run(&pool).await`.

4. Determine corpus sizes to test:
   - Default: `[100, 1000, 10000]`
   - If `--full`: also include `100000`
   - If `--corpus-size` specified: use only that value

5. Determine concurrency levels:
   - Default progression: `[10, 50, 100, 500]`
   - If `--concurrency` specified: use only that value

6. For each corpus size:
   a. Clear existing data: `corpus::clear_corpus(&pool).await`
   b. Seed corpus: `corpus::seed_corpus(&pool, size, num_projects).await`
   c. Build AppState with rate limits based on `--no-rate-limit` flag:
      - Create `PostgresMemoryStore::from_pool(pool.clone())`
      - Build `Config` with `rate_limit.enabled = !no_rate_limit`
      - Build `AppState` (embed_provider=None, embed_sender=None, metrics_handle from PrometheusBuilder::new().build_recorder().handle())
   d. Spawn test HTTP server on random port (reuse `api_test.rs` pattern):
      ```rust
      let api_routes = api::router(&state.config.rate_limit);
      let app = Router::new()
          .route("/health", get(health::status_handler))
          .merge(api_routes)
          .with_state(state);
      let listener = TcpListener::bind("127.0.0.1:0").await?;
      let addr = listener.local_addr()?;
      tokio::spawn(async move { axum::serve(listener, app).await.unwrap(); });
      let base_url = format!("http://{}", addr);
      ```
   e. Create reqwest::Client with pool_max_idle_per_host = concurrency.
   f. For each concurrency level:
      - Calculate total_ops: `cli.total_ops.unwrap_or(concurrency * 100)` (minimum 1000 ops)
      - Build LoadTestConfig
      - Run `client::run_workload(&config, &http_client).await`
      - Aggregate results: `metrics::aggregate_results(&results)`
      - Build LoadTestReport with:
        - `mode`: "raw" if no_rate_limit, "rate-limited" otherwise
        - `ops_per_sec`: total_ops / elapsed_secs
        - `error_rate`: total_errors / total_ops
        - `fly_tier_recommendation`: `report::recommend_fly_tier(concurrency, search_p95, error_rate)`
      - If baseline provided: `report::compare_baseline(&report, &baseline)`
      - Save report: `report::save_report(&report, json_path, md_path)`
      - Print summary to stdout: concurrency, ops/sec, p95, error_rate, tier recommendation

7. If `--save-baseline`: copy the last report JSON as `load_test_baseline.json` in output dir.

8. Print final summary: total runs completed, output directory.

**If `--no-rate-limit` is NOT set (default), also run with rate limits enabled:**
Run the same workload twice — once with `rate_limit.enabled = false` (raw capacity), once with `rate_limit.enabled = true` (production). Label reports accordingly. This produces paired "raw" vs "rate-limited" results per the locked decision.

**Anti-patterns to avoid:**
- Do NOT use `#[sqlx::test]` — the binary connects directly via `PgPool::connect`.
- Do NOT install a global Prometheus recorder — use `PrometheusBuilder::new().build_recorder()` for local recorder (avoids panic if metrics already initialized).
- Do NOT spawn unbounded tasks — always use the Semaphore in `run_workload`.
