<task type="auto">
  <name>Task 2: Corpus seeder with batch SQL inserts and pre-computed embeddings</name>
  <files>crates/memcp-core/src/load_test/corpus.rs</files>
  <action>
Create `crates/memcp-core/src/load_test/corpus.rs` that seeds a Postgres database with N memories + pre-computed embeddings for load testing.

Follow the `stress_test.rs` bulk insert pattern but adapted for the load test:

1. **`seed_corpus(pool: &PgPool, count: usize, num_projects: usize) -> Result<()>`**
   - Bulk insert `count` memories using raw SQL batch inserts (100 rows per statement, matching stress_test.rs).
   - Each memory gets:
     - `id`: `gen_random_uuid()::text`
     - `content`: Varied realistic content from a template pool (20+ templates covering code snippets, decisions, facts, preferences — richer than stress_test.rs's 20 items)
     - `type_hint`: Distributed as 40% fact, 25% preference, 20% decision, 15% instruction
     - `tags`: 30% none, 40% one tag, 30% two tags (from tag pool)
     - `project`: Distributed round-robin across `num_projects` project names ("project-0", "project-1", ..., "project-{N-1}") for multi-tenant simulation
     - `actor_type`: 'agent'
     - `audience`: 'global'
     - `embedding_status`: 'done' (critical — prevents embedding worker from interfering)
     - `created_at` / `updated_at`: `NOW()`
   - After memories inserted, batch insert rows into `memory_embeddings` table:
     - One row per memory: `memory_id`, `embedding` (random 384-dim unit vector via `rand` crate), `model_name` = 'load-test-synthetic'
     - Use random unit vectors (NOT zero vectors) so cosine similarity produces non-trivial results during search. Generate via `rand::thread_rng().gen::<[f32; 384]>()` then normalize to unit length.
   - Use `indicatif::ProgressBar` for progress during seeding (especially for 10k+ sizes).
   - Print summary at end: "{count} memories seeded across {num_projects} projects in {elapsed}s"

2. **`clear_corpus(pool: &PgPool) -> Result<()>`**
   - TRUNCATE `memory_embeddings` and `memories` tables (CASCADE). Used before each test run for clean state.

3. **Content template pool** (const array, 20+ entries):
   - Mix of: code pattern decisions ("Use async/await for all I/O operations"), user preferences ("Prefers dark mode in all editors"), architectural facts ("PostgreSQL tsvector enables full-text search"), project conventions ("Always use structured logging with request_id spans"), bug fix notes ("Fixed race condition in embedding pipeline by adding Semaphore"). Vary content length: some 1-line, some 3-4 lines.

Important anti-patterns to avoid (from RESEARCH.md):
- Do NOT set `embedding_status = 'done'` without also inserting into `memory_embeddings`. Both must exist for vector search to work.
- Do NOT use zero vectors — they produce degenerate cosine similarity (always 0.0). Use random unit vectors.
- Verify the Postgres trigger fires on raw INSERT (sets tsvector column). If it doesn't, run `UPDATE memories SET search_tsv = to_tsvector('english', content)` after batch insert.

<task type="auto">
  <name>Task 3: Report generation with JSON, Markdown, and baseline regression</name>
  <files>crates/memcp-core/src/load_test/report.rs</files>
  <action>
Create `crates/memcp-core/src/load_test/report.rs` for generating load test output.

Follow patterns from `src/benchmark/report.rs` (save_report/load_report).

1. **`generate_markdown_report(report: &LoadTestReport) -> String`**
   - Produces a markdown document with:
     - Header: "# Load Test Report — {timestamp}"
     - Config summary: corpus size, concurrency, R/W ratio, mode (raw/rate-limited), duration
     - Git SHA if available (from `git rev-parse --short HEAD`)
     - Per-endpoint table: `| Endpoint | Ops | Errors | p50 | p95 | p99 | Mean | Max |`
     - Overall summary: total ops/sec, error rate
     - Fly.io tier recommendation if present
     - Baseline regression section if regressions detected (table with endpoint, baseline p95, current p95, change%, flagged)

2. **`save_report(report: &LoadTestReport, json_path: &Path, md_path: &Path) -> Result<()>`**
   - Write `serde_json::to_string_pretty` to json_path
   - Write `generate_markdown_report` output to md_path

3. **`load_baseline(path: &Path) -> Result<Option<LoadTestReport>>`**
   - If file exists, deserialize and return Some. Otherwise None.

4. **`compare_baseline(current: &LoadTestReport, baseline: &LoadTestReport) -> RegressionReport`**
   - For each endpoint in current, find matching endpoint in baseline
   - Compute `change_pct = ((current_p95 - baseline_p95) as f64 / baseline_p95 as f64) * 100.0`
   - Flag if `change_pct > 20.0`
   - Return RegressionReport with baseline_date from baseline.timestamp

5. **`recommend_fly_tier(concurrency: usize, p95_search_ms: u64, error_rate: f64) -> String`**
   - Map concurrency + performance to Fly.io tier recommendation using the tier table from RESEARCH.md:
     - concurrency <= 10, p95 < 200ms, errors < 1% -> "Entry (shared-cpu-1x)"
     - concurrency <= 50 -> "Starter (shared-cpu-2x)"
     - concurrency <= 100 -> "Growth (shared-cpu-4x)"
     - concurrency <= 200 -> "Launch (performance-2x)"
     - concurrency <= 500 -> "Scale (performance-4x)"
     - else -> "Enterprise (performance-8x)"
   - Adjust upward if p95 > 500ms or error_rate > 5% at a given tier
   - Include Postgres connection limit note: "Requires Fly Postgres {tier} ({N} connection limit)"

6. **`get_git_sha() -> Option<String>`**
   - Run `git rev-parse --short HEAD` via `std::process::Command`. Return None on failure.
