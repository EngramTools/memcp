**IMPORTANT: Recorder installation caveat (Pitfall 6 from RESEARCH):** The global Prometheus recorder can only be installed once per process. Since `#[sqlx::test]` runs tests in the same process, use `std::sync::OnceLock` to guard recorder installation. All tests in this file share the same recorder — this is fine because Prometheus counters are additive and tests verify presence, not exact values.

**Test helper:**
```rust
use std::sync::OnceLock;
use metrics_exporter_prometheus::PrometheusHandle;

static RECORDER: OnceLock<PrometheusHandle> = OnceLock::new();

fn get_or_install_recorder() -> PrometheusHandle {
    RECORDER.get_or_init(|| {
        memcp::transport::metrics::install_prometheus_recorder()
    }).clone()
}
```

```rust
async fn spawn_test_server(state: AppState) -> String {
    let api_routes = api::router(&state.config.rate_limit)
        .layer(axum::middleware::from_fn(memcp::transport::metrics::metrics_middleware));
    let app = Router::new()
        .route("/health", get(memcp::transport::health::health_handler))
        .route("/status", get(memcp::transport::health::status_handler))
        .route("/metrics", get(memcp::transport::metrics::metrics_handler))
        .merge(api_routes)
        .with_state(state);
    let listener = tokio::net::TcpListener::bind("127.0.0.1:0").await.unwrap();
    let addr = listener.local_addr().unwrap();
    tokio::spawn(async move { axum::serve(listener, app).await.unwrap() });
    format!("http://{}", addr)
}
```

Create `crates/memcp-core/tests/rate_limit_test.rs` with integration tests for the rate limiting infrastructure.

**Test strategy:** Rate limiting is hard to test at sustained RPS in CI. Instead, configure extremely low limits (1 rps, burst_multiplier=1) so a burst of 2 rapid requests guarantees a 429.

**Test helper:** Same `make_test_state` and `spawn_test_server` as metrics_test.rs but with custom RateLimitConfig:

```rust
async fn make_rate_limited_state(pool: PgPool) -> AppState {
    let handle = get_or_install_recorder();
    let store = PostgresMemoryStore::from_pool(pool).await.unwrap();
    let mut config = Config::default();
    // Very low limits to trigger 429 in tests
    config.rate_limit.enabled = true;
    config.rate_limit.store_rps = 1;
    config.rate_limit.burst_multiplier = 1;
    AppState {
        ready: Arc::new(AtomicBool::new(true)),
        started_at: Instant::now(),
        caps: config.resource_caps.clone(),
        store: Some(Arc::new(store)),
        config: Arc::new(config),
        embed_provider: None,
        embed_sender: None,
        metrics_handle: handle,
    }
}
```

**Tests to write:**

1. `test_rate_limit_returns_429` — Configure store_rps=1, burst=1. Send 2 rapid POST /v1/store requests back-to-back (no delay). Second request should get HTTP 429. Verify the response body contains `"error"` field. Note: the first request consumes the single token; the second arrives before refill.

2. `test_rate_limit_429_has_retry_after_header` — Same setup as above. On the 429 response, verify the `Retry-After` header is present.

3. `test_rate_limit_429_has_json_body` — On the 429 response, parse the JSON body. Verify it contains `"error": "rate limited"` and `"retry_after_ms"` as a number.

4. `test_rate_limit_disabled_allows_all` — Configure `rate_limit.enabled = false`. Send many rapid requests. All should return non-429 status (200 or 4xx from validation, never 429).

5. `test_rate_limit_health_not_limited` — Configure very low global limits. Send many GET /health requests rapidly. None should return 429 (health endpoint is outside /v1/* and should not be rate-limited).

**Note on test flakiness:** Token bucket refill timing can cause flakiness. To mitigate:
- Use extremely low limits (rps=1, burst=1)
- Send requests as fast as possible (no delays)
- If a test is flaky, retry the burst pattern (send 3+ requests to guarantee at least one 429)
