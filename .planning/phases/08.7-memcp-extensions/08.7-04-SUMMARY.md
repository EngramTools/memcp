## What Was Built

**Tasks 1-3 were pre-implemented in commit 0f6d57b** as forward-work during the 08.7-01 phase. This plan added the final wiring layer (Task 4):

### IPC Protocol Extension (`ipc.rs`)

New `embed_multi` message type over the existing Unix domain socket:

- Request: `{"type": "embed_multi", "text": "query text"}`
- Response: `{"embeddings": {"fast": [...], "quality": [...]}}`

`start_embed_listener` gains an optional `(Arc<EmbeddingRouter>, Arc<PostgresMemoryStore>)` parameter. When provided, handles `embed_multi` by calling `router.embed_query_all_tiers(text, store)` which already implements the lazy optimization (skips non-default tiers with zero embeddings in corpus).

Single-model daemons receive `None` for this parameter and return an error response — CLI falls back to text-only search (fail-open).

### Daemon Wiring (`daemon.rs`)

At step 3.7, daemon passes `Some((router.clone(), store.clone()))` to `start_embed_listener` when `router.is_multi_model()` is true. Single-model daemons pass `None`.

### CLI Dual-Query Search (`cli.rs`)

`cmd_search` now branches on `config.embedding.is_multi_model()`:

- **Multi-model path**: calls `embed_multi_via_daemon()` → receives `HashMap<tier, Vec<f32>>` → converts to `HashMap<tier, pgvector::Vector>` → calls `hybrid_search_multi_tier()`
- **Single-model path**: existing `embed_via_daemon()` → `hybrid_search()` (backward compat)
- **Daemon offline (multi-model)**: calls `hybrid_search_multi_tier()` with empty tier map (BM25+symbolic only)
- **Daemon offline (single-model)**: existing degradation path

### Multi-Tier Recall (`postgres.rs`)

`recall_candidates_multi_tier(tier_embeddings, session_id, ...)`:

- Single tier: delegates to `recall_candidates()` directly
- Multi-tier: queries each tier's embedding separately, merges by best relevance score
- Result is deduped (one entry per memory_id), sorted by relevance desc, truncated to max_memories
