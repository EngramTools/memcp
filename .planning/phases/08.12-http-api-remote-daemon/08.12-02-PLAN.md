**Step 1: Add `--remote` global flag to Cli struct in main.rs**

```rust
struct Cli {
    #[command(subcommand)]
    command: Commands,

    #[arg(long, global = true)]
    skip_migrate: bool,

    /// Route commands through HTTP API instead of direct Postgres.
    /// Can also be set via MEMCP_URL environment variable.
    #[arg(long, env = "MEMCP_URL", global = true)]
    remote: Option<String>,
}
```

The `env = "MEMCP_URL"` attribute means clap reads from the env var automatically if --remote is not passed. This is the standard clap pattern.

**Step 2: Create `dispatch_remote()` in cli.rs**

Add a `pub async fn dispatch_remote()` helper. This is the single function used by all commands for HTTP dispatch:

```rust
/// Send a command to a remote memcp daemon via HTTP API.
///
/// Returns the parsed JSON response. Exits with error on non-success HTTP status.
pub async fn dispatch_remote(
    base_url: &str,
    command: &str,
    body: serde_json::Value,
) -> Result<serde_json::Value> {
    let client = reqwest::Client::builder()
        .timeout(std::time::Duration::from_secs(30))
        .build()?;
    let url = format!("{}/v1/{}", base_url.trim_end_matches('/'), command);
    let resp = client.post(&url)
        .header("content-type", "application/json")
        .json(&body)
        .send()
        .await
        .map_err(|e| anyhow::anyhow!("Remote request to {} failed: {}", url, e))?;

    let status = resp.status();
    let body_text = resp.text().await
        .map_err(|e| anyhow::anyhow!("Failed to read remote response: {}", e))?;

    let json: serde_json::Value = serde_json::from_str(&body_text)
        .unwrap_or_else(|_| serde_json::json!({"error": body_text}));

    if !status.is_success() {
        let msg = json.get("error")
            .and_then(|v| v.as_str())
            .unwrap_or("remote error");
        anyhow::bail!("Remote memcp error ({}): {}", status.as_u16(), msg);
    }

    Ok(json)
}
```

Key design choices:
- Single reqwest::Client per call is fine (CLI is short-lived, no need for connection pooling).
- 30-second timeout covers `wait: true` store operations.
- Parse response as text first, then as JSON — handles non-JSON error responses gracefully.
- On non-success status, extract error message from JSON body and bail with it.

**Step 3: Thread `remote` through command dispatch in main.rs**

In main.rs, after parsing, thread `cli.remote` into each command dispatch. The simplest pattern: pass `remote: Option<String>` as a new first parameter to each `cmd_*` function, OR check it before calling cmd_*:

```rust
// In the match on cli.command:
Commands::Recall { query, session_id, reset, workspace, first, limit, boost_tags } => {
    if let Some(ref remote_url) = cli.remote {
        let body = json!({
            "query": query,
            "session_id": session_id,
            "first": first,
            "reset": reset,
            "workspace": workspace,
            "limit": limit,
            "boost_tags": boost_tags,
        });
        let result = cli::dispatch_remote(remote_url, "recall", body).await?;
        println!("{}", serde_json::to_string_pretty(&result)?);
    } else {
        // existing local path
        let store = ...;
        cli::cmd_recall(&store, &config, ...).await?;
    }
}
```

Apply this pattern to ALL 5 data commands: recall, search, store, annotate, update.

For each command, build the JSON body using `serde_json::json!()` with the same field names as the API request types from Plan 01. Use `#[serde(skip_serializing_if = "Option::is_none")]` semantics by only including non-None fields. The `json!()` macro handles this naturally — `"field": null` is fine (the API uses `serde(default)` and treats null/missing the same).
