# Phase 08.12 Plan 01: HTTP API Server Summary

**One-liner:** REST API on port 9090 with 6 /v1/* routes (recall, search, store, annotate, update, status) using in-process embedding тАФ no IPC hop for daemon-local callers.

## What Was Built

Expanded the existing health HTTP server (port 9090) with a full REST API layer for core memcp operations. The OpenClaw plugin and CLI remote mode can now call memcp over HTTP without needing Postgres credentials or a memcp binary.

### AppState (transport/health/mod.rs)

`HealthState` renamed to `AppState` with three new fields:
- `config: Arc<Config>` тАФ full daemon config for handlers (RecallConfig, StoreConfig, etc.)
- `embed_provider: Option<Arc<dyn EmbeddingProvider + Send + Sync>>` тАФ in-process embedding for recall/search
- `embed_sender: Option<mpsc::Sender<EmbeddingJob>>` тАФ pipeline sender for store wait=true polling

### transport/api module (7 new files)

| File | Route | Notes |
|-|-|-|
| `mod.rs` | router() | merges into health server |
| `types.rs` | тАФ | 5 request structs, error_json helper |
| `recall.rs` | POST /v1/recall | queryless + query-based, first=true preamble injection |
| `search.rs` | POST /v1/search | hybrid search + salience re-ranking, field projection |
| `store.rs` | POST /v1/store | resource caps, temporal extraction, wait=true polling |
| `annotate.rs` | POST /v1/annotate | delegates to annotate_logic() shared helper |

### daemon.rs wiring

Pipeline creation moved before health server spawn so `embed_sender` is available. AppState constructed with full config + embedding provider + pipeline sender.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing] EmbeddingPipeline created before health server**
- **Found during:** Task 2 (daemon.rs wiring)
- **Issue:** Plan placed health server spawn at step 2.7 (before pipeline creation at step 4), but AppState needs `embed_sender` which requires the pipeline
- **Fix:** Moved pipeline creation (steps 3, 3.5, 3.7, 4) to execute before health server spawn. Reordered sections in daemon.rs. Functional outcome identical тАФ pipeline still created before backfill and main loop
- **Files modified:** crates/memcp-core/src/transport/daemon.rs

**2. [Rule 1 - Simplification] Store handler uses polling not oneshot for wait=true**
- **Found during:** Task 2 implementation
- **Issue:** Plan suggested oneshot channel on EmbeddingJob for wait=true. However, store handler enqueues via `embed_sender.try_send()` which creates a new EmbeddingJob тАФ the existing oneshot completion_tx pattern requires the pipeline worker to send back on the channel. Since the test state has no embed_sender, and polling already works (CLI uses it), polling is simpler and more robust
- **Fix:** Used polling (200ms intervals, sync_timeout_secs timeout) matching CLI cmd_store behavior exactly
- **Impact:** No behavioral difference тАФ same timeout, same response shape

**3. [Rule 3 - Unused code] Removed dead struct types from health/mod.rs**
- **Found during:** Task 1 (cargo check produced dead_code warnings)
- **Issue:** StatusResponse, ComponentHealth, ResourceInfo, ResourceUsage structs became unused when status_handler was converted to return `Json<serde_json::Value>` for the pub alias pattern
- **Fix:** Removed unused structs, kept inline serde_json::json!() in status_handler
