# Phase 08.12 Plan 02: CLI Remote Mode Summary

**One-liner:** CLI transparently routes recall/search/store/annotate/update through HTTP API when `MEMCP_URL` is set, via single `dispatch_remote()` helper in cli.rs.

## What Was Built

Added `--remote <url>` / `MEMCP_URL` support to all 5 data CLI subcommands. When the env var or flag is set, the CLI becomes a thin HTTP client — no Postgres connection needed.

### Task 1: --remote global flag and dispatch_remote() helper

**`crates/memcp/src/main.rs`** — Added `remote: Option<String>` field to `Cli` struct with `env = "MEMCP_URL"` and `global = true`. Added inline remote dispatch branches for all 5 data commands: `recall`, `search`, `store`, `annotate`, `update`. Each branch builds a `serde_json::json!()` body matching the API request shapes from Plan 01, calls `cli::dispatch_remote()`, and pretty-prints the result.

**`crates/memcp-core/src/cli.rs`** — Added `pub async fn dispatch_remote(base_url, command, body)`:
- Builds `reqwest::Client` with 30s timeout
- POSTs to `{base_url}/v1/{command}` with JSON body
- Reads response as text first (handles non-JSON errors gracefully)
- On non-success status: extracts `error` field from JSON body and bails with status code

### Task 2: End-to-end remote mode tests

Extended `crates/memcp-core/tests/api_test.rs` with 4 new tests:

| Test | What it validates |
|-|-|
| `test_dispatch_remote_store` | HTTP POST returns JSON with `id` field (UUID) |
| `test_dispatch_remote_recall_queryless` | Response has `session_id` + `memories` array |
| `test_dispatch_remote_error_handling` | 503 (not ready) produces Err containing "503" |
| `test_dispatch_remote_invalid_url` | Connection refused surfaces as connection error |

All 15 api_test tests pass (11 from Plan 01 + 4 new).
