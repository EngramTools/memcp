# Phase 08.11.1 Plan 01: Bi-temporal Search Event-Time Wiring Summary

**Temporal search boost now uses `event_time.unwrap_or(created_at)` so a memory about "in 2019" stored today correctly ranks higher on "2019 memories" queries.**

## Accomplishments

- Wired `event_time` into the temporal soft boost at step 12.5 in `server.rs` — `hit.memory.event_time.unwrap_or(hit.memory.created_at)` replaces bare `hit.memory.created_at`
- Added explanatory comment clarifying the bi-temporal rationale (event time = content reference time, not store time)
- Added 3 unit tests proving: (1) event_time present + in range → boost; (2) event_time absent → fallback to created_at; (3) event_time out of range → no boost even when created_at is in range

## Files Created/Modified

- `crates/memcp-core/tests/unit/temporal.rs` — Added `make_memory()` helper, `bi_temporal_in_range()` helper replicating the boost logic, and 3 test functions covering all bi-temporal cases

## Decisions Made

- `event_time.unwrap_or(created_at)` — event_time takes strict precedence when present. A memory about past events with an out-of-range event_time gets no boost even if its created_at is in range. This is the correct bi-temporal semantic.
- Unit tests validate the `unwrap_or` selection logic via a local helper function rather than calling server internals directly. This is sufficient because the match arms are unchanged and the inline nature of the boost means extraction would require a full refactor.
