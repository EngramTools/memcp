# Phase 24.75 Plan 01: Migration 028 Orchestrator + DDL Summary

One-liner: Two-step migration 028 apparatus built and dry-run-verified — Rust orchestrator binary re-embeds parents in short transactions with A1 + dimension guardrails, followed by a DDL-only SQL migration that drops parent_id, chunk_index, total_chunks, and the parent_id index idempotently.

## What Was Built

### 1. `crates/memcp-core/migrations/028_drop_chunks.sql` (10 lines, DDL-only)

Drops `idx_memories_parent_id` first, then `ALTER TABLE memories DROP COLUMN IF EXISTS` for `parent_id`, `chunk_index`, `total_chunks`. No transaction wrapper (sqlx::migrate! owns it). Per Research R-1, any async embedding work lives in the orchestrator binary, not here. Idempotent per CONTEXT.md D-02.

### 2. `crates/memcp-core/src/bin/migrate_028_collapse_chunks.rs` (438 lines)

`required-features = ["local-embed"]`. clap-parsed CLI with `--database-url`, `--dry-run`, `--report-path`. Flow:

1. Load config → create embedding provider via `memcp::daemon::create_embedding_provider`.
2. Pitfall 5 guard: `model_dimension(provider.model_name())` vs `provider.dimension()` — bail before any UPDATE if mismatched.
3. Idempotency probe: `SELECT DISTINCT parent_id FROM memories WHERE parent_id IS NOT NULL AND deleted_at IS NULL`. Zero rows → exit 0 with "no-op" message.
4. `--dry-run` → print parent IDs, exit 0.
5. Per-parent loop (indicatif progress bar):
   - `store.get_chunks_by_parent(parent_id)` — last use before Plan 03 deletes the helper.
   - `store.get(parent_id)` — fetch parent row.
   - A1 guardrail: bail if `parent.content.is_empty()` — would destroy data.
   - `detect_and_reassemble(&parent, &chunks)` returns `parent.content` on A1-CONFIRMED/UNDECIDABLE-EMPTY; reassembles with header stripping on A1-REFUTED.
   - `provider.embed(&full_content)` outside any transaction.
   - Short `pool.begin()` → optional `UPDATE memories SET content` (only if reassembled) → `UPDATE memory_embeddings SET embedding = $1::vector, updated_at = NOW() WHERE memory_id = $2 AND is_current = true` → `tx.commit()`.
   - Append JSON line to `data/migration_028_report.jsonl` (parent_id, chunk_count_collapsed, new_embedding_dim, reassembled flag, timestamp).
6. Final sweep: one `DELETE FROM memories WHERE parent_id IS NOT NULL` statement outside any embedding call.

## A1 Branch Taken

`A1-UNDECIDABLE-EMPTY` (from `24.75-A1-PROBE.md`). Decision ladder inside `detect_and_reassemble`:

1. `chunks.is_empty()` → return `parent.content` unchanged. Zero-chunk parents are steady-state rows; orchestrator won't touch them (idempotency probe already filters to parents with chunks, so this branch only fires if chunks disappear between the probe and the helper call — e.g., racing delete).
2. If `parent.content.len() + (chunks.len() * 40) >= chunk_total`, parent holds full content (A1-CONFIRMED). Return `parent.content`.
3. Otherwise (A1-REFUTED) reassemble: for each chunk in `chunk_index` order, strip leading `[From: "...", part N/M]\n` context-header line, concatenate.

Additionally, before `detect_and_reassemble` runs, the main loop asserts `!parent.content.is_empty()` — the A1 runtime guardrail recommended by 24.75-A1-PROBE.md. An empty parent with chunks violates A1 and aborts the migration cleanly.

## Deviations from Plan

### Interface substitutions (verified public paths)

The plan's skeleton code referenced `memcp::embedding::create_embedding_provider`. The canonical public path is `memcp::daemon::create_embedding_provider` (re-exported at `memcp::daemon::` via `lib.rs::pub use transport::daemon`). Used the real path. No API change.

### Store construction

Plan skeleton showed `PostgresMemoryStore::new(pool.clone())` taking a prebuilt pool. The actual constructor is `PostgresMemoryStore::new(database_url: &str, run_migrations: bool)`. Chose to construct two separate entities: a direct `PgPool` for write transactions (so begin/commit scopes stay inside the binary), and a `PostgresMemoryStore` for the read helpers (`get`, `get_chunks_by_parent`). Cost: an extra pool of ~10 warm connections for the store. Benefit: no need to add a pool-sharing constructor to the store just for a one-shot binary.

### Migration 028 SQL — comment wording

Initial draft comment said "embedding is async network I/O". The acceptance criterion checks `grep -c "embedding"` must be 0. Rephrased to "vectorization is async network I/O" — same meaning, zero matches. No functional change.

### Integration test shim

Plan said the test should exercise orchestrator helper logic "via the public entry points exposed by this binary." Rust integration tests live in a separate crate and cannot import symbols from `src/bin/*.rs` (binary crates are not library targets). Re-declared `detect_and_reassemble` as a small shim inside the test with a comment explaining the lockstep requirement. The binary's inline unit tests are the canonical coverage; the integration shim is a tripwire — if either side diverges without the other, both suites will ping. This preserves the plan's intent (A1 branches get end-to-end integration-test coverage) without a public-API-shaped wart on the binary.
