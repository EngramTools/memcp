# Phase 24.75 Plan 04: Runtime Topic-Span Drill-Down Summary

Delivered the agent-facing `get_memory_span(memory_id, topic)` tool across MCP, HTTP, and CLI, all delegating to a single shared compute function. CHUNK-04 closed — long memories now get topic-precise drill-down on demand instead of forcing pre-computed chunks into search results.

## MCP tool description (agent-first review)

```
Drill into a long memory: returns the span semantically closest to `topic`.
Splits the memory on the fly (no stored chunks), embeds each candidate span,
returns the best match with byte offsets. First call on a given memory is
slow (embedding N spans); topic embeddings are cached per session. Use when
a whole-memory hit is too long to fit in context — ask for the topic-specific
span instead of retrieving the full memory.
Params: memory_id (required UUID), topic (required, describe what you are
looking for in words, max 512 chars).
Returns: {content: string (the span), span: {start: int (byte offset), end: int}}.
```

The description is self-sufficient per CLAUDE.md agent-first rule — no system-prompt hints required. It spells out the latency characteristic, the cache semantics, the specific use case ("whole-memory hit too long to fit in context"), and the exact return shape.

## Shared helper signature

```rust
pub async fn compute_memory_span(
    store: Arc<dyn MemoryStore + Send + Sync>,
    provider: Arc<dyn EmbeddingProvider + Send + Sync>,
    topic_cache: Arc<Mutex<HashMap<String, Vec<f32>>>>,
    memory_id: &str,
    topic: &str,
) -> Result<MemorySpanResponse, MemcpError>
```

Returns `MemorySpanResponse { content: String, span: SpanBytes { start, end } }`. MCP + HTTP + CLI produce byte-identical output for the same `(memory_id, topic)` pair — one code path, no drift risk.

## Cache bounds

- `topic_embedding_cache`: `Arc<Mutex<HashMap<String, Vec<f32>>>>` cap **100 entries** on both `AppState` (HTTP) and `MemoryService` (MCP). When full, drops a single arbitrary entry (`HashMap::iter().next()`) before inserting. Stored at session scope — reset on daemon restart.
- `MAX_SPANS_PER_MEMORY = 64`: hard cap against DOS on pathologically long memories (T-24.75-04-02).
- `TOPIC_MAX_LEN = 512`: topic string length cap (T-24.75-04-01).
- `SPAN_MAX_CHARS = 2048`, `SPAN_OVERLAP_SENTENCES = 1`, `SPAN_MIN_CONTENT_CHARS = 256`: query-time chunking params hardcoded so the tool behaves identically regardless of operator config (ChunkingConfig is vestigial post-24.75).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Splitter's `chunk_content` prefixes a context header that breaks byte-offset mapping**

- **Found during:** Task 1, first test run.
- **Issue:** Plan specified using `chunk_content` which emits `[From: "topic", part N/M]\n` prefix on each span. That prefix is NOT a substring of `memory.content`, so `memory.content.find(span.content)` fails and offsets drift.
- **Fix:** Switched to `splitter::split_sentences` directly (returns raw `Vec<Vec<String>>` without the header). Each winning span's byte offsets are computed by anchoring the first + last sentence back to `memory.content`, then returning `memory.content[start..end]` verbatim — guarantees byte-exact invariant.
- **Files modified:** `crates/memcp-core/src/transport/api/memory_span.rs`
- **Commit:** 9d5bde1

**2. [Rule 3 - Blocking] AppState struct literal rebuilt in 6 test helpers + load_test.rs**

- **Found during:** Task 1, build-fail cascade after adding `topic_embedding_cache` field.
- **Issue:** Every file that constructs `AppState { ... }` as a struct literal (not via a constructor) had to be extended with the new field. Plan mentioned api_test.rs only; missed ingest_test.rs, rate_limit_test.rs (×2), metrics_test.rs, trust_retrieval_test.rs, load_test.rs (×2), daemon.rs.
- **Fix:** Added `topic_embedding_cache: Arc::new(Mutex::new(HashMap::new()))` to all 8 instances.
- **Commit:** 9d5bde1

**3. [Rule 3 - Blocking] MemoryService pulled `HashMap<String, Vec<f32>>` field requires the existing `std::collections::HashMap` import**

- **Found during:** Task 2.
- **Issue:** Adding `topic_embedding_cache` field to MemoryService uses `HashMap<String, Vec<f32>>` — the server.rs file already imports `HashMap` at line 7 for UuidRefMap. Reused.
- **Commit:** c38eef2

### Plan-spec adjustments

- **compute_memory_span signature:** Uses `MemcpError` instead of `anyhow::Error` as per actual codebase conventions. Handler maps NotFound / Validation variants to HTTP codes directly.
- **`store.get(id)` returns `Memory`, not `Option<Memory>`:** Plan sketched an `Option` branch; actual trait returns `Result<Memory, MemcpError>` with a `NotFound` variant. `?` propagation handles it; MCP/HTTP handlers do pattern-matching on MemcpError::NotFound for uniform 404 response.
- **`embed_provider` is the AppState field name (not `embedding_provider`):** corrected throughout.
