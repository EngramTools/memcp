# Phase 11.2 Plan 01: Trust-Weighted Retrieval Scoring Summary

**Trust-weighted composite scoring multiplying salience by trust_level in both MCP and HTTP search paths, with 4 integration tests**

## Performance

- **Duration:** 18 min

- **Completed:** 2026-03-09T05:21:00Z

## Accomplishments
- Composite score formula changed from `0.5 * norm_rrf + 0.5 * norm_sal` to `0.5 * norm_rrf + 0.5 * (norm_sal * trust)` across all search paths
- LLM re-ranking blend updated from `0.7 * llm + 0.3 * sal` to `0.7 * llm + 0.3 * (sal * trust)` in MCP server path
- 4 integration tests validating trust demotion, zero-trust edge case, backward compatibility, and ordering
- Zero new dependencies

## Files Created/Modified

- `crates/memcp-core/src/transport/server.rs` - Trust multiplier in composite scoring + LLM re-ranking (MCP path)
- `crates/memcp-core/src/transport/api/search.rs` - Trust multiplier in composite scoring (HTTP API path)

## Decisions Made
- Trust multiplies only the salience component (not RRF) -- retrieval relevance stays independent of trust level. A trust=0.0 memory's composite = `0.5 * norm_rrf + 0` (only RRF contribution).
- HTTP API search handler (api/search.rs) has its own composite scoring, separate from MCP server.rs. Both updated for uniform trust-weighted behavior.
- Tests bypass content-hash dedup by inserting via raw SQL, allowing identical-content memories with different trust_level.
